## Het Probleem

Als fanatieke gamer heb ik één groot probleem: games zijn duur. Maar het echte probleem is niet de prijs op zich — het is dat diezelfde game vandaag €59,99 kost op Steam, maar volgende week voor €15,99 te koop is op Humble Bundle, en weer een week later in de Steam sale voor €19,99. En dan realiseer je je dat je hem drie maanden geleden al op GOG voor €12 had kunnen krijgen.

Platforms zoals [IsThereAnyDeal](https://isthereanydeal.com) en [SteamDB](https://steamdb.info) bestaan al, maar ik wilde begrijpen hoe zo'n systeem werkt. Niet alleen gebruiken, maar **bouwen**. Hoe verzamel je prijzen van 30+ verschillende winkels? Hoe monitor je prijsgeschiedenis? Hoe stuur je automatisch alerts wanneer een game in de aanbieding is?

Dit was mijn kans om een full-stack platform te bouwen dat real-world problemen oplost — en tegelijk te leren werken met externe APIs, async backend architectuur, en complexe frontend state management.

## De Oplossing

IsThereADeal is een game prijsvergelijkingsplatform dat prijzen van 30+ winkels verzamelt en real-time monitort. Gebruikers kunnen:

- 🔍 **Prijzen vergelijken** van Steam, GOG, Humble, GreenManGaming, Fanatical en 25+ andere officiële winkels
- 💰 **Key reseller prijzen** bekijken (opt-in) van G2A, Eneba en AllKeyShop
- 📊 **Prijsgeschiedenis** bekijken met interactieve grafieken (2 jaar historische data)
- ⭐ **Wishlist** aanmaken met doelprijzen
- 🔔 **Prijsalerts** ontvangen via email wanneer games onder je doelprijs komen
- 📈 **Trending deals** ontdekken gesorteerd op beste deals

Het platform is volledig responsive, heeft een donker gaming theme, en is geoptimaliseerd voor snelle laadtijden.

## De Technische Uitdaging

Het bouwen van een prijsvergelijker klinkt simpel, maar de uitdagingen komen al snel:

### 1. **Data van Meerdere Bronnen Aggregeren**

Games hebben verschillende IDs op verschillende platforms:
- Steam gebruikt `appid` (bijv. 220 voor Half-Life 2)
- IsThereAnyDeal gebruikt een `plain` identifier (bijv. "halflife2")
- CheapShark heeft weer een eigen `gameID` systeem

**Oplossing:** Ik bouwde een ID mapping service die Steam appids automatisch vertaalt naar ITAD en CheapShark IDs. Deze mappings worden 24 uur gecached om API calls te besparen.

### 2. **Performance: 13+ Seconden Laadtijd**

Initieel laadde een game pagina in 13+ seconden. Waarom?
- 5+ externe API calls per game (Steam, ITAD, CheapShark, key resellers)
- Elk met timeouts van 10-15 seconden
- Geen caching, alles realtime
- Key resellers altijd geladen (8+ seconden extra)

**Oplossing: Multi-layer Caching & Optimalisatie**

**Backend:**
- In-memory response cache met 5 minuten TTL
- Connection pooling (pool size 10, max overflow 20)
- Optimized timeouts per API (Steam 8s, CheapShark 6s, ITAD 5-6s)
- Key resellers opt-in: alleen laden op verzoek
- 24h cache voor ID mappings

**Frontend:**
- TanStack Query met 10 minuten staleTime
- 30 minuten garbage collection
- Automatic retry op gefaalde requests

**Resultaat:**
- **Nieuwe games:** 5-8 seconden *(was 13+ seconden)*
- **Cached games:** Instant *(in-memory cache)*
- **Previously loaded:** 1-2 seconden *(database cache)*

### 3. **Real-time Prijsalerts**

Hoe check je duizenden alerts zonder de API rate limits te raken?

**Oplossing: Smart Alert Checking**
- Vercel Cron job draait dagelijks om 09:00
- Haalt alleen actieve alerts op (niet getriggerde)
- Batch verwerking: 50 alerts per batch
- Controleert huidige prijs vs doelprijs
- Stuurt email via aiosmtplib (async SMTP)
- Markeert alert als triggered + timestamp

```python
# Backend alert checker (vereenvoudigd)
@router.post("/run-check")
async def check_alerts():
    alerts = await db.get_active_alerts()
    
    for alert in alerts:
        current_price = await get_lowest_price(alert.game_id)
        
        if current_price <= alert.target_price:
            await send_email_alert(alert)
            alert.triggered_at = datetime.now()
            await db.save(alert)
```

### 4. **Prijsgeschiedenis Visualisatie**

ITAD API geeft prijshistorie per winkel over 2 jaar. Dat zijn honderden datapunten die gevisualiseerd moeten worden.

**Oplossing: Recharts met Smart Filtering**
- Multi-line chart met één lijn per winkel
- Kleurcodering per winkel
- Tooltip toont alle prijzen op geselecteerde datum
- Responsive chart die schaalt op mobiel
- Decimals keurig afgekapt op 2 decimalen

## Geleerde Lessen

### 1. **Caching is Koning**
Zonder caching: 13+ seconden. Met multi-layer caching: 1-2 seconden voor cached requests. Connection pooling alleen scheelde al 40% laadtijd.

### 2. **Opt-in > Opt-out**
Key resellers initieel altijd laden was een slechte keuze. Na opt-in maken: 60% sneller voor meeste gebruikers.

### 3. **User Feedback is Essentieel**
Loading skeletons, toast notifications, password strength indicators — dit soort UX details maken het verschil tussen "works" en "feels good to use".

### 4. **Async is Complex maar Waard**
Async Python (SQLAlchemy, httpx, aiosmtplib) heeft een leercurve, maar de performance gains zijn enorm. Vooral met connection pooling.

### 5. **External APIs Falen**
Timeouts, rate limits, downtime — externe APIs falen constant. Defensief programmeren met fallbacks en error handling is niet optioneel.

## De Uitdaging: Steam Wishlist Import

Een feature die nog niet af is: **Steam wishlist import**. Klinkt simpel, maar:
- Steam heeft geen officiële API voor private wishlists
- Je moet scrapen met BeautifulSoup, wat fragiel is
- Of de gebruiker moet hun Steam API key delen (security risk)
- Of je moet Steam OAuth implementeren (complex)

**Huidige plan:** Steam OAuth met read-only permissions. Maar dit vereist Steam Developer registratie en API approval, wat weken kan duren.
